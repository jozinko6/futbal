# KOMPLETNÝ PROMPT PRE PRESTAVBU FUTBALOVEJ HRY

Pracuješ priamo v existujúcom repozitári:

https://github.com/jozinko6/futbal

Aktuálna vetva `main` obsahuje rozpracovanú browserovú futbalovú hru postavenú na:

* Next.js,
* React,
* TypeScript,
* Canvas 2D,
* fixed timestep simulácii,
* vlastnej fyzike,
* AI,
* Socket.IO multiplayeri,
* prezentačnej vrstve s particles, camera shake a ball trail.

Aktuálny projekt formálne obsahuje množstvo funkcií a testov, ale výsledok je stále divný, nekonzistentný a prakticky nehrateľný.

Tvojou úlohou je:

1. najskôr opraviť všetky fundamentálne chyby gameplayu,
2. zjednodušiť ovládanie a simuláciu,
3. vybudovať jeden skutočne hrateľný offline zápas,
4. následne premeniť hru na originálneho duchovného nástupcu arkádového futbalu v štýle Taito Cup Finals,
5. čo najvernejšie zachytiť jeho tempo, kameru, mierku postáv, ovládanie, atmosféru, režimy, výber kapitána a špeciálne strely,
6. nepoužiť žiadne ukradnuté alebo ripnuté assety.

Nepýtaj sa na potvrdenie po každej etape.

Postupuj automaticky po menších kontrolovateľných krokoch.

Nevytváraj nový projekt od nuly, pokiaľ nie je konkrétny modul objektívne jednoduchšie nahradiť. Zachovaj použiteľné časti existujúcej architektúry, ale neboj sa odstrániť zlé systémy.

# ETAPA 2 – JEDEN ZDROJ PRAVDY PRE LOPTU

Aktuálne sa vlastníctvo lopty eviduje súčasne cez:

* `ball.ownerId`,
* `player.hasBall`.

Tieto hodnoty sa rozchádzajú a vytvárajú neplatné stavy.

Odstráň `player.hasBall` ako samostatne zapisovaný zdroj pravdy.

Použi iba:

```ts
ball.ownerId
```

Vytvor helper:

```ts
export function playerHasBall(
  state: MatchState,
  playerId: number,
): boolean {
  return state.ball.ownerId === playerId;
}
```

Nikde priamo nenastavuj:

```ts
player.hasBall = true;
player.hasBall = false;
```

Ak je pole potrebné pre kompatibilitu alebo serializáciu, aktualizuj ho iba v jednej centralizovanej synchronizačnej funkcii. Gameplay logika ho nesmie čítať.

Vytvor explicitný stav lopty:

```ts
type BallMode =
  | 'FREE'
  | 'CONTROLLED'
  | 'PASS'
  | 'SHOT'
  | 'AERIAL'
  | 'GK_HELD'
  | 'RESTART'
  | 'OUT_OF_PLAY';
```

BallState musí obsahovať:

```ts
interface BallState {
  mode: BallMode;
  ownerId: number | null;
  previousOwnerId: number | null;
  lastTouchPlayerId: number | null;
  lastTouchTeam: Team | null;
  controlStartedTick: number;
  releaseTick: number;
}
```

Zaveď invarianty:

* `CONTROLLED` vždy vyžaduje platného `ownerId`,
* `GK_HELD` vždy vyžaduje brankára,
* `FREE`, `PASS`, `SHOT` a `AERIAL` nesmú mať majiteľa,
* hráč nemôže začať prihrávku ani strelu, ak nie je majiteľom lopty,
* po kontaktnom ticku sa vlastníctvo okamžite zruší,
* počas wind-up zostáva lopta pod kontrolou hráča,
* `resolvePossession` nesmie znovu prideľovať už kontrolovanú loptu.

Pridaj runtime invariant kontrolu v development režime.

Pri porušení invariantov vypíš presnú chybu vrátane:

* ticku,
* ball mode,
* ownerId,
* aktívneho hráča,
* aktuálnej akcie.

# ETAPA 3 – PREPRACOVANIE KONTROLY LOPTY

Realistický systém voľnej lopty počas každého driblingového dotyku odstráň.

Pre arkádový futbal použi dva jasné režimy:

## CONTROLLED

Keď hráč kontroluje loptu:

* lopta zostáva spoľahlivo pri hráčovi,
* neodpojí sa náhodne počas obyčajného behu,
* jej pozícia sa vizuálne mení podľa fázy driblingovej animácie,
* strieda ľavú a pravú nohu,
* pri šprinte je mierne ďalej pred hráčom,
* stále je však súčasťou kontrolovaného stavu.

Použi foot socket:

```ts
interface BallControlSocket {
  forwardOffset: number;
  lateralOffset: number;
  foot: 'left' | 'right';
}
```

Výsledná poloha môže byť vypočítaná z:

* facing hráča,
* rýchlosti,
* aktuálneho animačného frame,
* dribble phase,
* smeru zmeny pohybu.

Nepoužívaj generické `resolvePossession()` na odobratie lopty majiteľovi len preto, že sa vizuálna lopta na chvíľu dostala mimo malého control radius.

## FREE BALL

Lopta sa stane voľnou iba keď nastane:

* prihrávka,
* strela,
* úspešný tackle,
* blok,
* odraz od hráča,
* odraz od brankára,
* zlé spracovanie veľmi prudkej prihrávky,
* kolízia so žrďou,
* aut,
* špeciálna herná udalosť.

## SPRACOVANIE PRIHRÁVKY

Bežná krátka prihrávka má byť spracovaná spoľahlivo.

Zlý prvý dotyk používaj iba keď:

* prihrávka je veľmi prudká,
* lopta prichádza zo zlého uhla,
* hráč je pod okamžitým tlakom,
* ide o vysokú loptu,
* hráč práve šprintuje opačným smerom.

Normálne prihrávky nesmú pravidelne náhodne odskočiť.

Cieľom je arkádová konzistentnosť, nie simulácia technických chýb.

# ETAPA 4 – OPRAVA INPUT LOOPU

Aktuálny InputManager môže spotrebovať edge udalosť v render frame, v ktorom neprebehne žiadny simulačný tick.

Tým sa môžu stratiť:

* prihrávky,
* strely,
* prepnutia hráča,
* high pass,
* dotykové tlačidlá.

Zmeň input architektúru.

Oddeľ:

```ts
ContinuousInput
EdgeInput
```

ContinuousInput:

```ts
interface ContinuousInput {
  moveX: number;
  moveY: number;
  aimX: number;
  aimY: number;
  sprintHeld: boolean;
  shootHeld: boolean;
  modifierHeld: boolean;
}
```

EdgeInput:

```ts
interface EdgeInput {
  passPressed: boolean;
  shootPressed: boolean;
  shootReleased: boolean;
  tacklePressed: boolean;
  switchPressed: boolean;
  specialPressed: boolean;
}
```

Edge udalosti uchovávaj v queue až do najbližšieho simulačného ticku.

Volanie inputu musí prebehnúť vnútri fixed-step slučky:

```ts
while (accumulator >= FIXED_DT) {
  const input = inputManager.consumeForSimulationTick();
  step(state, input, FIXED_DT);
  accumulator -= FIXED_DT;
}
```

Ak prebehne viac catch-up tickov:

* edge udalosť dostane iba prvý tick,
* continuous state dostanú všetky tick-y.

Pridaj test, ktorý simuluje:

* render pri 144 FPS,
* simuláciu pri 60 Hz,
* veľmi krátke stlačenie tlačidla.

Prihrávka sa nesmie stratiť.

# ETAPA 5 – ANALÓGOVÝ A DIGITÁLNY POHYB

Veľkosť analógového vstupu musí ovplyvňovať rýchlosť.

Implementuj response curve:

```ts
const magnitude = clamp(length(moveX, moveY), 0, 1);

if (magnitude < 0.20) {
  speed = 0;
} else if (magnitude < 0.45) {
  speed = walkSpeed;
} else if (magnitude < 0.78) {
  speed = jogSpeed;
} else {
  speed = runSpeed;
}
```

Pridaj:

* gamepad deadzone,
* touch joystick deadzone,
* response curve,
* voliteľný digitálny režim pre klávesnicu.

Pre klasický arkádový preset môže byť pohyb rýchlejší a digitálnejší.

Pre moderný preset zachovaj jemnú analógovú kontrolu.

## OTOČENIE

Odstráň opakované násobenie rýchlosti hodnotou `sharpTurnPenalty` každý tick.

Namiesto toho:

* vypočítaj rozdiel aktuálneho velocity direction a požadovaného smeru,
* zníž dostupné zrýchlenie podľa uhla,
* jednorazovo aplikuj malý brake impulse pri otočení väčšom než 120°,
* nepoužívaj geometrické násobenie rýchlosti každý tick.

Požadované správanie:

* 45° zmena smeru je rýchla,
* 90° zmena mierne spomalí,
* 180° otočenie trvá krátku, ale čitateľnú chvíľu,
* hráč sa nesmie zaseknúť,
* hráč musí okamžite reagovať na nový vstup.

## STAMINA

V hlavnom arkádovom režime vypni komplikovanú staminu.

Použi iba ľahký systém:

* hráč môže šprintovať stále,
* dlhý šprint mierne zníži maximálnu rýchlosť,
* po krátkom spomalení sa rýchlo obnoví,
* stamina nesmie hráča zablokovať pod konkrétnym prahom.

Kapitán môže mať lepšiu výdrž.

# ETAPA 8 – ACTION SYSTÉM A ANIMÁCIE

Každá akcia musí mať:

* wind-up,
* contact,
* recovery.

Simulácia rozhoduje o kontaktnom ticku.

Renderer iba zobrazuje výsledok.

Počas akcie použi locomotion multiplier:

```ts
windupMovementMultiplier
contactMovementMultiplier
recoveryMovementMultiplier
```

Odporúčanie:

* short pass wind-up: 0.55,
* shot wind-up: 0.35,
* contact: 0.05,
* recovery: 0.35 až 0.7,
* slide: vlastná rýchlosť a smer.

Pohyb nesmie prepisovať stav `shoot` alebo `pass`.

Animáciu vyberaj podľa:

```ts
player.currentAction?.type
player.currentAction?.phase
```

Nie iba podľa všeobecného `player.state`.

Pridaj mapovanie:

* shortPass,
* longPass,
* lobPass,
* normalShot,
* powerShot,
* superShot,
* hyperShot,
* standingTackle,
* shoulderCharge,
* slideTackle,
* flyingTackle,
* receive,
* header,
* goalkeeperCatch,
* goalkeeperDive,
* goalkeeperParry.

Oprav `slide`, aby nepoužíval fallback walk animáciu.

# ETAPA 9 – PRIHRÁVKY

Prihrávky musia byť rýchle, čitateľné a odpúšťajúce.

Implementuj:

* short pass,
* driven pass,
* through pass,
* lob pass,
* automatic one-touch pass.

## TARGETING

Pri stlačení prihrávky:

1. vezmi smer vstupu,
2. nájdi spoluhráčov v smere,
3. vypočítaj pass lane safety,
4. zohľadni pohyb príjemcu,
5. vyber najlogickejší cieľ,
6. ak žiadny cieľ nie je vhodný, prihraj do priestoru.

Pass lane safety musí kontrolovať:

* vzdialenosť obrancu od segmentu prihrávky,
* čas lopty na dosiahnutie bodu,
* čas obrancu na zachytenie,
* pohyb obrancu,
* pohyb príjemcu.

Prihrávka nesmie byť zablokovaná neviditeľnou logikou.

Musí dôjsť k fyzickému alebo jasne definovanému kontaktu obrancu s dráhou lopty.

## ONE-TOUCH

Ak hráč stlačí prihrávku tesne pred príchodom lopty:

* vykoná one-touch pass,
* nemusí najprv plne prevziať vlastníctvo,
* kontaktný tick sa vypočíta podľa príchodu lopty.

# ETAPA 10 – STREĽBA

# ETAPA 11 – OBRANA

Implementuj jasný kontextový obranný systém.

## SHOULDER CHALLENGE

* pri behu vedľa súpera,
* odtlačí súpera,
* môže uvoľniť loptu,
* malá pravdepodobnosť faulu pri správnom uhle.

## STANDING TACKLE

* krátky dosah,
* rýchla akcia,
* nízky recovery,
* používa sa zpredu alebo zboku.

## POKE TACKLE

* automatická kontextová varianta standing tackle,
* zasiahne iba loptu,
* minimálny kontakt s hráčom.

## SLIDE TACKLE

* samostatné tlačidlo,
* pevný smer po začatí,
* aktívna kontaktná fáza,
* výrazný recovery pri minutí.

## ARCADE TACKLES

Do režimu Arcade Cup môžeš pridať originálne prehnané animácie:

* jumping block,
* flying clearance,
* dramatic shoulder hit.

Nesmú však pôsobiť ako náhodná bitka a nesmú odoberať kontrolu na dlhý čas.

## FOUL TRACKING

Odstráň WeakMap z deterministickej simulácie.

Kontaktné dáta ukladaj do serializovateľného stavu hráča alebo akcie:

```ts
interface TackleContactState {
  startedTick: number;
  ballContactTick: number | null;
  playerContactTick: number | null;
  contactAngle: number;
  relativeSpeed: number;
}
```

To musí fungovať:

* offline,
* online,
* po reconnecte,
* v replayi,
* v testoch.

## VÝBEH

Pri výbehu nestačí nastaviť iba `aiTarget`.

Okamžite vypočítaj desired velocity smerom k intercept pointu.

Brankár musí fyzicky bežať k lopte.

## ZÁKROK

Počas dive musí mať aktívne zóny:

* hands,
* torso,
* feet.

Kolízia lopty s aktívnou save zónou môže vytvoriť:

* catch,
* parry,
* punch,
* deflection.

Brankára počas `goalkeeperDive` nevynechávaj zo všetkých kontaktov s loptou.

Použi predikciu trajektórie:

* miesto priesečníka s bránkovou čiarou,
* výška lopty,
* čas do kontaktu,
* reakčný čas,
* aktuálna pozícia brankára.

Brankár nesmie:

* reagovať na stlačenie tlačidla pred kopom,
* teleportovať sa,
* chytať loptu mimo dosahu,
* ignorovať loptu počas animácie dive.

Kapitánsky alebo boss brankár môže mať lepšie reakcie, ale stále musí rešpektovať fyzický dosah.

# ETAPA 17 – KAMERA V ŠTÝLE ARKÁDOVÉHO AUTOMATU

Aktuálnu top-down kameru nahraď v hlavnom režime pseudo-perspektívnou side-scrolling kamerou.

Herný pohľad:

* diagonálny pohľad z tribúny,
* horizontálne posúvanie za loptou,
* menší vertikálny pohyb,
* veľké postavy,
* viditeľná časť ihriska,
* bránka sa zobrazí pri priblížení k šestnástke.

Použi world coordinates nezávislé od render projekcie.

Vytvor projection layer:

```ts
interface ProjectedPoint {
  screenX: number;
  screenY: number;
  scale: number;
  depth: number;
}
```

Príklad princípu:

```ts
screenX = worldX - cameraX;
screenY = horizonY + worldY * perspectiveScaleY;
scale = lerp(farScale, nearScale, normalizedDepth);
```

Presné hodnoty nalaď podľa vizuálnej referencie.

Požiadavky:

* hráči bližšie k spodnej strane obrazovky sú mierne väčší,
* vzdialenejší hráči sú mierne menší,
* mierka sa nesmie meniť agresívne,
* nohy hráčov určujú depth sorting,
* tiene zostávajú na zemi,
* lopta má správnu výšku pri centri a strele,
* kamera nesmie pumpovať zoom,
* kamera nesmie trhať pri auto-switchi.

Použi camera deadzone.

Kamera sleduje primárne loptu, nie aktívneho hráča.

Aktívny hráč ovplyvňuje focus iba mierne.

Odstráň dvojité aplikovanie camera shake.

Neposúvaj priamo reálny camera state kvôli shake.

Použi:

```ts
renderCamera = {
  x: simulationCamera.x + shake.x,
  y: simulationCamera.y + shake.y,
};
```

Odstráň starý `cam.shake` systém používajúci `Math.random()` v `cameraOrigin`, ak je nahradený PresentationManagerom.

Shake aplikuj presne raz.

# ETAPA 26 – OPRAVA PREZENTAČNÝCH EFEKTOV

Aktuálne sa silný efekt spúšťa podľa hodnoty `power > 300`, hoci power je uložený v px/s. Tým môžu aj obyčajné prihrávky spúšťať Power Shot efekty.

interface BallKickedEvent {

# ETAPA 28 – TESTY FUNDAMENTÁLNEJ HRATEĽNOSTI

Pridaj prísne testy.

## INPUT

* krátky tap pri 144 FPS sa nestratí,
* edge event sa spracuje presne raz,
* držané tlačidlo nevytvára opakované pass eventy.

## POSSESSION

* `CONTROLLED` vždy má ownerId,
* voľná lopta nikdy nemá ownerId,
* hráč bez lopty nemôže strieľať,
* po kick contact ticku je ownerId null,
* wind-up zachováva vlastníctvo,
* žiadny desync po 100 000 tickoch.

## DRIBBLE

* bežný beh nespôsobí náhodnú stratu lopty,
* šprint nespôsobí stratu bez tackle alebo chyby,
* vizuálny offset lopty neovplyvní vlastníctvo,
* prudká zmena smeru zostáva ovládateľná.

## MOVEMENT

* 25 % analógový vstup je pomalší než 100 %,
* otočenie o 180° nezastaví hráča na sériu tickov,
* contain neintegruje pozíciu dvakrát.

## ACTIONS

* pass contact nastane presne raz,
* shot contact nastane presne raz,
* pohyb neprepíše shot animáciu,
* slide používa správny action state.

## SWITCHING

* auto-switch najviac raz za cooldown,
* manuálne prepnutie sa okamžite neprepíše,
* po prijatí lopty sa aktivuje majiteľ.

## GOALKEEPER

* dive môže fyzicky zasiahnuť loptu,
* lopta mimo save zóny nie je chytená,
* rush nastaví velocity smerom k intercept pointu,
* parry ponechá loptu živú.

## AI

* tím má maximálne jedného pressera,
* cover assignment sa neprepíše individuálnou AI,
* marking assignment nepoužíva troch obrancov na jeden cieľ bez dôvodu,
* AI neprihráva cez jasne blokovanú dráhu.

## PREZENTÁCIA

* short pass nespustí Power Shot shake,
* Power Shot spustí správny event,
* shake sa aplikuje iba raz.

# DEFINITION OF DONE

Projekt je dokončený až keď:

1. Krátke stlačenia tlačidiel sa nestrácajú.

2. Lopta má presne jeden zdroj vlastníctva.

3. Neexistuje stav, v ktorom hráč „má loptu“ a ball.ownerId ukazuje niečo iné.

4. Dribling je spoľahlivý a arkádový.

5. Analógový joystick ovláda intenzitu pohybu.
